# Provenance Guard — Demo Walkthrough Script

Target length: **~2–3 minutes**. Keep it short and unpolished — it's a quick tour, not a polished ad. The detailed evidence lives in the README; this is just you driving the system end-to-end and naming a few design decisions.

## Before you hit record (60-second dry run)

1. `python app.py` in one terminal (key is in `.env`).
2. In a second terminal, run the four curls below **once** so you're warm and you know what scores come back this session.
3. **Heads-up on the LLM signal:** Groq is non-deterministic — the AI essay usually lands `likely_ai` but can dip to `uncertain` on a bad roll. If that happens on a take, either re-run it or lean into it ("notice the system would rather say *uncertain* than wrongly accuse — that's the design"). Don't fight it.

Have these open on screen: the running server, the terminal for curls, and `planning.md` / `README.md` to glance at.

---

## Scene 1 — What it is (~20s)

> "This is **Provenance Guard** — a backend a creative platform plugs in to tell readers whether a post was written by a person or generated by AI. The point isn't to police anyone — it's attribution and trust. And critically, it never pretends to be certain, because AI detection isn't a solved problem."

Show `planning.md` Architecture diagram for two seconds while you say it.

---

## Scene 2 — A submission, end to end (~40s)

Run the AI example and read the response out loud:

```bash
curl -s -X POST http://localhost:5000/submit \
  -H "Content-Type: application/json" \
  -d '{"text": "Artificial intelligence represents a transformative paradigm shift in modern society. It is important to note that while the benefits of AI are numerous, it is equally essential to consider the ethical implications. Furthermore, stakeholders across various sectors must collaborate to ensure responsible deployment.", "creator_id": "demo-ai"}'
```

> "Every submission runs through **two independent signals** — an LLM that reads it semantically, and pure-Python stylometry that measures structure like sentence-length variance. Here you can see both scores in the response, combined into one confidence, and a plain-language label. This one comes back **likely AI**."

Point at `signals.llm_score`, `signals.stylo_score`, `confidence`, and `label` in the JSON.

---

## Scene 3 — The three labels (~30s)

Run the human one:

```bash
curl -s -X POST http://localhost:5000/submit \
  -H "Content-Type: application/json" \
  -d '{"text": "ok so i finally tried that new ramen place downtown and honestly? underwhelming. the broth was fine but they put WAY too much sodium in it and i was thirsty for like three hours after. probably wont go back unless someone drags me there", "creator_id": "demo-human"}'
```

> "Casual human writing comes back **likely human**, with a clearly different score."

Then a borderline one:

```bash
curl -s -X POST http://localhost:5000/submit \
  -H "Content-Type: application/json" \
  -d '{"text": "Our innovative platform leverages cutting-edge technology to deliver seamless solutions. Designed with the user in mind, it empowers teams to unlock their full potential.", "creator_id": "demo-mid"}'
```

> "And when the signals disagree or the evidence is weak, it returns **uncertain** instead of guessing. That uncertain band is wide *on purpose* — labeling a human's work as AI is the worst outcome on a writing platform, so the AI threshold is deliberately stricter than the human one."

---

## Scene 4 — Appeal (~25s)

Grab a `content_id` from the AI submission and appeal it:

```bash
curl -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "PASTE-AI-CONTENT-ID", "creator_reasoning": "I wrote this essay myself for a class."}'
```

> "If a creator thinks they were misclassified, they appeal. The status flips to **under review** and their reasoning is logged right next to the original decision — no automatic re-classification, a human looks at it."

---

## Scene 5 — Audit log + rate limit (~30s)

```bash
curl -s http://localhost:5000/log
```

> "Every decision and appeal is in a structured audit log — both signal scores, the confidence, the status. There's the appealed entry showing `under_review`."

Then show the rate limiter (separate terminal, server running):

```bash
for i in $(seq 1 12); do
  curl -s -o /dev/null -w "%{http_code}\n" -X POST http://localhost:5000/submit \
    -H "Content-Type: application/json" \
    -d '{"text": "rate limit test submission for the demo.", "creator_id": "flooder"}'
done
```

> "And `/submit` is rate-limited — ten a minute. A real creator never hits that, but a script flooding the endpoint does. First ten return 200, then it's 429s."

---

## Scene 6 — Close (~15s)

> "So: two signals, honest confidence, three clear labels, an appeal path, and a full audit trail. The whole design is built around one idea — when you can't be sure, say so, and give the creator a way to push back."
